# 酒馆网页聊天隔离与沉浸式界面设计

日期：2026-04-29
项目：`C:\Users\kanye\tavern-web`

## 1. 背景

当前项目已经完成两轮能力：

- D：网页操作指令面板、一键复制、阶段话术模板
- E1（本地已完成，待远端收尾）：网页聊天发送、聊天接口、WS 实时同步、房间关闭后禁发

但当前 E1 仍有两个问题：

1. 网页聊天与公开时间线混在一起，旁观页也可能看到玩家网页发言
2. `room` / `spectator` 页面仍偏说明卡片，不够像真正可玩的聊天软件

本设计用于先收尾 E1，再继续把网页体验重构为更像“主持人带桌”的沉浸式聊天界面。

## 2. 已确认目标

### 2.1 本轮顺序

1. 先收尾 E1：远端同步、重启服务、生产验证
2. 再继续做下一轮体验升级

### 2.2 下一轮主方向

- 采用“剧情主持人沉浸风”而不是 QQ/Discord 纯模仿风
- `room` 页作为参与者主界面
- `spectator` 页作为只读实况页
- 底部采用“发言框 + 推荐话术并列”，其中发言框更大

### 2.3 聊天可见性边界

用户确认：网页聊天应当“只给本局参与者看，旁观页看不到”。

第一版将“参与者”定义为：

- 能访问 `room` 页的用户视为房内参与者视图用户
- 暂不做账号级、成员级的强身份校验
- 先完成页面级/通道级隔离，后续再考虑更强权限模型

## 3. 非目标

本轮不做以下内容：

- 网页直接控制 QQ 机器人
- 房主 PIN / 管理口令
- 账号绑定或成员身份认证
- 不同参与者各自看到不同私聊内容
- 暴露任何私密身份、角色映射或角色行动信息

## 4. 方案选择

比较过三种方案：

1. 仅前端隐藏旁观聊天
2. 公开时间线 / 参与者聊天 双轨制
3. 引入更强身份系统的私域聊天

最终选择方案 2。

### 选择理由

- 真正隔离参与者聊天与旁观信息
- 数据结构清晰，便于后续扩展房主控制台或更强权限
- 可以在不引入账号系统的前提下显著提升可玩性

## 5. 数据与接口设计

### 5.1 数据分层

保留 `room_timeline` 作为公开时间线，只存以下内容：

- 阶段切换
- 主持人播报
- 当前事件
- 下一步提示
- 房间开始 / 房间关闭

新增 `room_chat_messages` 作为参与者聊天流，字段建议为：

- `id`
- `room_id`
- `user_name`
- `content`
- `created_at`

### 5.2 API 设计

#### `GET /api/rooms/:roomId/spectator`

返回：

- `room`
- `presentation`
- `public_timeline`

不返回 `members`，旁观页人数统一使用 `room.member_count`，房主展示统一使用 `room.owner_name`。

不得返回参与者聊天。

#### `GET /api/rooms/:roomId/room-view`

返回：

- `room`
- `members`
- `presentation`
- `public_timeline`
- `participant_chat`

用于 `room` 页首屏一次取全。

#### `POST /api/rooms/:roomId/chat`

语义：发送到本局参与者聊天。

返回：

- `ok`
- `chat_message`

错误规则保持清晰：

- `404 room_not_found`
- `409 room_closed`
- `400 invalid_chat_message`

### 5.3 WebSocket 设计

为现有订阅机制增加频道概念：

- `channel=room`
- `channel=spectator`

广播规则：

- 公开更新：同时发给 `room` 与 `spectator`
- 参与者聊天：只发给 `room`
- 房间关闭：同时发给 `room` 与 `spectator`

## 6. 前端界面设计

### 6.1 `room` 页

`room` 页重构为参与者主界面：

1. 顶部状态条：剧本、房间号、阶段、进度、当前行动者、关闭状态
2. 中间主聊天流：
   - 主持人播报
   - 公开系统事件
   - 参与者聊天
3. 底部输入区：
   - 大发言框
   - 推荐话术快捷填入
   - 明确按钮文案“发送到本局聊天”
4. 次级操作区：复制命令、阶段模板收纳为侧栏或折叠区

页面文案需要明确：

> 这里是本局参与者聊天，不会显示到旁观页。

### 6.2 `spectator` 页

`spectator` 页重构为只读实况页：

- 保持沉浸风格
- 只显示主持人播报与公开事件
- 不显示参与者聊天
- 不提供聊天输入框
- 保留低权重辅助信息，不抢主视觉

## 7. 错误处理与状态边界

需要明确处理以下边界：

1. 房间关闭：首屏和运行中都要禁用输入区
2. 发送失败：针对 400 / 404 / 409 给出明确提示
3. 重复消息：聊天消息带 `id`，前端按 `id` 去重
4. WS 断连：自动重连并重新订阅，必要时回退拉聚合接口

## 8. 测试方案

### 8.1 接口测试

- `spectator` 不返回参与者聊天
- `room-view` 返回参与者聊天
- `POST /chat` 写入参与者聊天表
- 房间关闭后返回 `409`

### 8.2 WS 测试

- `room` 收到公开更新和参与者聊天
- `spectator` 只收到公开更新
- `spectator` 收不到参与者聊天

### 8.3 UI E2E

至少覆盖：

1. `room` 页发送聊天
2. 另一 `room` 页实时收到
3. `spectator` 页看不到该聊天
4. 阶段推进时两端都看到公开更新
5. 房间关闭后输入区禁用
6. 刷新后聊天与公开记录都能恢复

- 刷新后记录仍在
